# Sagittarius — Project Instructions for Spacecraft Commander

This file is the authority for how SC-Commander operates inside this repo.
Global agent rules, skill definitions, and communication protocol live in
`~/.config/opencode/AGENTS.md`. This file adds project-specific routing,
conventions, tooling, and discipline.

## SC-Commander Integration

### Resolver

The `.space/` directory tracks active missions. Before any mutating or
mission-scoped work, resolve the current mission:

```bash
scripts/spacecraft resolve --json
```

Block if resolver safety is not `safe` or no mission is selected. Direct the
user to `scripts/spacecraft missions` then `scripts/spacecraft use <id>`.

### Lane Detection

Route work automatically based on intent:

| Intent | Lane | Entry Point |
|--------|------|-------------|
| ask, consult, research, explain | Advisory | Direct response |
| add, build, implement, feature | Mission | `/sc-start` → full gate chain |
| fix, debug, diagnose, error | Debug | `/sc-debug` → 5-step discipline |
| human edits, config, small fix, docs | Quick | `/sc-quick` → branch → self-review → report ready |

### Gate Discipline

Mission lane requires every gate. Never skip:

```
spec → plan → build(→verify→commit) → review → ship
```

- **spec**: Written in mission artifacts; must exist before `/sc-plan`.
- **plan**: Generated by `sc-planner` subagent; ≤7 verifiable tasks.
- **build**: Each task implemented by `sc-coder`, verified by `sc-tester`.
- **review**: `sc-reviewer` subagent inspects diff, evidence, and readiness.
- **ship**: Merge to `main` only after all gates pass.

### Agent Delegation

These are the only subagent types for this project. Never delegate project code
to a general-purpose agent:

| Subagent | Capability | Use For |
|----------|-----------|---------|
| `sc-coder` | Write | Frontend (React/TS), backend (Rust), config, scripts |
| `sc-tester` | Write | Unit tests, component tests, e2e, browser QA scripts |
| `sc-planner` | Read | Decomposing spec into executable plan |
| `sc-reviewer` | Read | Inspecting diff, evidence, and release readiness |
| `sc-designer` | Read | UI direction, critique, anti-slop review |

**Writing product code directly is forbidden.** Always delegate to `sc-coder`.
**Writing tests directly is forbidden.** Always delegate to `sc-tester`.

## Product Context

Sagittarius is a travel planning cockpit for group trips. The public brand is
**Joii**. "Sagittarius" is the local project codename — never expose it in
public-facing UI, landing pages, or user-visible copy.

The core experience is an operational workspace, not a marketing site. Prefer
calm, scan-friendly, work-focused UI that helps organizers compare plans, edit
itinerary details, track members, and resolve trip decisions quickly.

### 6-Phase Journey Model (v0.7.0+)

The app guides users through six phases. Each phase has curated left-rail
navigation, context rail rules, and view layout. The system auto-derives the
current phase from trip data.

1. **Dreamer** — destination imagery, seasonal cards, trip creation
2. **Flexible Hunter** — date window, budget ranges
3. **Route Builder** — MapLibre map, waypoint pins, route lines
4. **Detail Planner** — Smart Itinerary Table (source of truth), plan variants
5. **Group Wrangler** — members, invites, polls, RSVPs, expense settlement
6. **On-Trip Companion** — mobile-first, Now/Next card, offline-ready

On-Trip Companion has no left rail (mobile bottom nav instead). Dreamer,
Flexible Hunter, Route Builder, and On-Trip Companion suppress the right
context rail.

### Itinerary Path Model

Itinerary items support activity-level path planning with overlapping plan
variants:

- `main` is the default path.
- Overlapping activities can be assigned to generated day plans (`Plan A`, `Plan B`, etc.).
- Day-level filtering and trip-level filtering are separate concerns.
- Import/export and automatic overlap resolution must preserve the path model
  instead of flattening activities back into one list.
- `pathGroupId`, `pathId`, `pathName`, and `pathRole` are data contract fields —
  keep them in sync between UI, import/export, API patches, and tests.

## Frontend Conventions

### Stack

- **Framework**: Next.js 16 (App Router), React 19
- **Language**: TypeScript 6.0, strict mode
- **Styling**: Tailwind CSS 4.2, PostCSS
- **Testing**: Vitest 4.1 (unit + Storybook browser), Playwright
- **Runtime**: bun 1.3.5
- **Component catalog**: Storybook 10.4

### Styling Rules

Use existing Tailwind utilities, component classes, design tokens, and
Storybook patterns first. Add custom CSS only after confirming the current
Tailwind/design-system vocabulary cannot express the behavior clearly.

For Tailwind utilities whose value is a single CSS variable token, use the
canonical shorthand form: `text-(--color-text-muted)`, `bg-(--color-surface)`,
`border-(--color-border)`, `rounded-(--radius-md)`. Reserve bracketed arbitrary
values for expressions that are not a single token: `rgb(...)`, `color-mix(...)`,
gradients, fallback vars, shadows, and arbitrary properties.

Keep cockpit surfaces dense but readable. Avoid landing-page patterns inside
the workspace: oversized hero composition, decorative card stacks,
gradient-only sections, or explanatory in-app marketing copy.

### Component Patterns

When adding controls:
- Use existing `Button`, `Icon`, i18n messages, and table/dialog patterns.
- Keep table and toolbar controls stable across desktop and mobile.
- Check that context rail, table scroll, and mobile viewport behavior do not
  block itinerary toolbar actions.

### TypeScript

Strict mode is on. No `any` without justification. Prefer explicit return
types on exported functions. Path aliases: `@/*` → `./*` (configured in
tsconfig).

## API, State & Backend

### Dual Mode

The app can run in local mode and API mode. Any itinerary mutation that
changes activity paths must handle both:

- **Local mode**: update the trip draft through the local commit/history flow.
- **API mode**: patch backend items with current `expectedVersion` values and
  handle `version_conflict` by reloading the latest cockpit before retrying or
  recomputing local placement.

### Data Boundary

A data boundary (`tripRepository` / `tripDataAdapter`) isolates UI code from
whether data comes from local storage or backend APIs. Never let components
import backend types or call fetch directly for trip data.

### Backend (Rust)

- **Language**: Rust 1.95, edition 2024
- **Database**: PostgreSQL, migrations in `backend/migrations/`
- **API**: REST + WebSocket (to be built after frontend)
- **Crate**: `sagittarius-api` under `backend/crates/`

### Key Backend Tables

`trips`, `itinerary_items`, `plan_variants`, `plan_checks`, `expenses`,
`expense_reminders`, `trip_tasks`, `trip_daily_briefings`, `booking_docs`,
`trip_join_invite_tokens`, `account_vault_items`, `users`,
`auth_attempt_locks`, `trusted_devices`, `place_geocode_cache`.

## Tests & QA

### Required verification after any implementation task

At minimum: `make frontend-verify` (lint + typecheck + unit tests + storybook +
build). If the change touches itinerary paths, also run browser QA relevant to
the affected surfaces.

### Key test files for itinerary/path changes

- `frontend/src/trip/itinerary-paths.test.ts`
- `frontend/src/trip/itinerary.test.ts`
- `frontend/src/features/itinerary/components/SmartItineraryTable.test.tsx`
- `frontend/src/trip/workspace/sagittarius-app/SagittariusApp.test.tsx`

### Browser QA

When a user-facing feature touches the itinerary table, verify at least one
real browser flow for the visible control, context rail interaction,
console/page errors, and mobile overflow.

### QA viewport checklist

- 320px — mobile
- 768px — tablet
- 1024px — small desktop
- 1440px — full desktop

### Visual QA checklist

- No page-level horizontal scroll.
- Table scroll contained to table viewport.
- Sidebar expanded/collapsed transitions.
- Right rail relocation on tablet.
- Readable Thai and English mixed text.
- No overlapping controls or clipped labels.

## Git & Branch Conventions

### Branch naming

```
feature/<description>     # Mission work
fix/<description>         # Debug fixes
docs/<description>        # Documentation-only changes
chore/<description>       # Config, tooling, refactors
```

### Commit conventions

Follow Conventional Commits: `type(scope): description`. Types: `feat`, `fix`,
`docs`, `chore`, `refactor`, `test`, `style`. Target 1–3 commits per branch,
max 5. Squash WIP before merge.

### Merge discipline

Rebase on `main` → verify gates pass → `git merge --no-ff`. Never force-push
shared branches. Never commit on `main` directly.

## Completion & Checkpoint Discipline

After every implementation task:
1. `sc-tester` captures fresh verification evidence (command output, test
   results, screenshots).
2. Commit the finished work with a conventional commit message.
3. Mark the task complete.

Only leave changes uncommitted when the user explicitly asks not to commit or
when verification is blocked and the remaining risk needs user direction.

No evidence = not done. Every task must include concrete, freshly captured
verification output before it is reported complete.

## Deployment

Whenever changes are merged to `main`, production deployment must run
automatically through the production deployment path:

```bash
make production-deploy-gate    # env check + release signoff
```

Followed by production publish steps in `docs/production-docker-cloudflare.md`
and GitHub workflow expectations. Manual merge-to-production approval does not
replace this automatic deploy requirement.

For production route changes, teams must complete a browser smoke check
immediately after deploy in this order: `/trips`, map route, auth flow.

## Issue Tracking

When an unfixed project issue is discovered during a turn:
1. Create a GitHub issue with context, evidence, impact, suggested fix path,
   and verification criteria.
2. Use `issues.md` only as a local index to GitHub Issues and archived audit
   records.
3. When fixed, close the GitHub issue with the fix summary and verification
   evidence.
