Implementation Plan: Trip Sheets

Goal: เพิ่ม named Trip Sheet ใน trip เดียว โดย reuse planVariants และไม่ชนกับ activity paths.

Architecture: UI copy ใช้ “Trip Sheet”; internal model ยังเป็น PlanVariant. Sheet create/switch เปลี่ยน activePlanVariantId; activity path fields ยังอยู่ใน itinerary rows.

Tech Stack: Next.js, React 19, TypeScript, Vitest, existing Rust API routes for plan variants.

Current vs Proposed

Current

Current: whole-trip variants exist, but itinerary UI does not expose them as named sheets.

Proposed

Proposed: visible Trip Sheet control drives the existing active variant; path model stays nested.

new selector/create UI, shared planVariants reused, risk avoid mixing sheets with activity filters.

Implementation Section

Acceptance Criteria

  • Current Trip Sheet name appears in itinerary workspace.
  • Organizer can create a named Trip Sheet; viewer/traveler cannot.
  • Creating a sheet switches the active sheet.
  • Switching a sheet updates visible itinerary rows by planVariantId.
  • New itinerary stops created after sheet switch/create use the current sheet planVariantId.
  • Imported rows apply to the current sheet and preserve activity path fields.
  • Activity path controls and fields remain unchanged inside the selected sheet.
  • Local mode and API mode both support create and switch.
  • API version conflict during sheet publish/switch reloads latest cockpit state.

Freeze this contract before parallel work. UI worker and App worker must use these exact names.

The table owns only draft input state for the new sheet name. API/local pending state and errors live in SagittariusApp.
